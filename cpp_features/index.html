C++ Engine Features

Everything the multi-threaded DPI engine can do — and how to use it.

🛡️ Firewall Mode PRODUCTION-READY

The C++ engine can function as a Layer 7 application firewall. Unlike traditional firewalls that only filter by IP/port, the DPI engine inspects packet payloads to identify which application generated the traffic — then blocks or forwards accordingly.

It physically drops packets that match blocking rules and writes only forwarded traffic to the output PCAP. With raw socket integration, it can sit inline on a network path and filter traffic in real time.

🏢 Enterprise Filtering Block YouTube, TikTok, Netflix on a corporate network by app name alone
🔒 Parental Controls Block specific domains with wildcard matching — *.tiktok.com
📊 Traffic Shaping Identify bandwidth-heavy apps and selectively drop or throttle
🕵️ Threat Detection Detect and drop connections to known malicious IPs or domains
bash — firewall mode 
# Block YouTube + Facebook + a suspicious IP in one command
./dpi_engine live_capture.pcap filtered.pcap \
    --block-app YouTube \
    --block-app Facebook \
    --block-domain tiktok.com \
    --block-ip 10.0.0.99

# Result: only "clean" traffic makes it to filtered.pcap

⚡ Multi-Threaded Pipeline

🧵 Thread Architecture NOT IN WEB VERSION

The engine runs a multi-stage pipeline: PCAP Reader → Load Balancers → Fast Path Workers → Output Writer, each as a separate thread. This means packet reading, inspection, and output writing all happen concurrently.

bash — configure threads 
# Default: 2 LBs × 2 FPs each = 4 worker threads
./dpi_engine input.pcap output.pcap

# Scale up: 4 LBs × 4 FPs = 16 worker threads
./dpi_engine input.pcap output.pcap --lbs 4 --fps 4

🔗 Consistent Hashing Load Balancing NOT IN WEB VERSION

Load balancers use FiveTupleHash (src IP, dst IP, src port, dst port, protocol) to route all packets from the same connection to the same Fast Path worker. This guarantees flow affinity without any shared state between workers.

🔒 Thread-Safe Queues with Backpressure NOT IN WEB VERSION

Each inter-thread queue (TSQueue<T>) uses mutex + condition variables with a configurable max size (default: 10,000). When a queue is full, the producer blocks until space is available — preventing memory blowup under heavy load.

📊 Atomic Statistics NOT IN WEB VERSION

All counters (total packets, forwarded, dropped, TCP/UDP counts) use std::atomic<uint64_t> for lock-free thread-safe updates. Per-app stats use a separate mutex-protected map.

🔍 Deep Packet Inspection

🔐 TLS SNI Extraction

Parses TLS ClientHello messages to extract the Server Name Indication (SNI) field — the plaintext hostname sent before encryption starts. This is how the engine identifies HTTPS traffic without decryption. Works on port 443.

🌐 QUIC SNI Extraction NOT IN WEB VERSION

QUIC (HTTP/3) uses UDP:443 instead of TCP. The engine parses QUIC Initial packets which contain the TLS ClientHello inside CRYPTO frames, extracting SNI in the same way. This catches traffic from Chrome, YouTube, and Google services that default to QUIC.

📄 HTTP Host Header Extraction NOT IN WEB VERSION

For unencrypted HTTP (port 80), the engine reads the Host: header from the HTTP request to identify the destination website. Combined with SNI extraction, this covers both HTTP and HTTPS traffic.

🗃️ DNS Query Extraction NOT IN WEB VERSION

Extracts queried domain names from DNS request packets (port 53). This provides an additional classification signal — even before the actual connection is established, the engine knows what domain the client is trying to reach.

🚫 Blocking & Filtering Rules

🌍 IP Blocking

Block all traffic from a specific source IP address. The IP is parsed into a uint32_t and checked in an unordered_set for O(1) lookup speed.

bash 
./dpi_engine input.pcap output.pcap --block-ip 192.168.1.50 --block-ip 10.0.0.5

📱 Application Blocking

Block entire applications by name. The engine first classifies the traffic using SNI/Host/DNS, then applies the blocking rule. Supported apps include: YouTube, Facebook, Netflix, Google, GitHub, TikTok, Twitter, Instagram, LinkedIn, Spotify, and more.

bash 
./dpi_engine input.pcap output.pcap --block-app YouTube --block-app Netflix

🔗 Domain Blocking with Wildcards NOT IN WEB VERSION

Block domains with substring matching and wildcard support. *.facebook.com blocks all Facebook subdomains including m.facebook.com, static.facebook.com, etc.

bash 
./dpi_engine input.pcap output.pcap --block-domain facebook.com --block-domain tiktok

🚪 Port Blocking NOT IN WEB VERSION

Block all traffic to specific destination ports. For example, block port 8080 to prevent proxy access, or block port 22 to prevent SSH tunneling.

📋 Block Reason Reporting NOT IN WEB VERSION

When a packet is blocked, the engine records exactly why — was it an IP rule, an app rule, a domain rule, or a port rule? The BlockReason struct contains the type and a human-readable detail string.

💾 Rule Persistence NOT IN WEB VERSION

Save your blocking rules to a file with saveRules() and reload them later with loadRules(). Deploy the same ruleset across multiple instances or environments.

🔌 Connection Tracking

🗂️ Per-Thread Flow Tables NOT IN WEB VERSION

Each Fast Path worker maintains its own unordered_map<FiveTuple, FlowEntry>. Because flow affinity guarantees all packets from the same connection go to the same worker, these tables are thread-local — no locks needed for flow lookup or update.

🔄 TCP State Machine NOT IN WEB VERSION

The engine tracks TCP connection state transitions: NEW → SYN_SENT → ESTABLISHED → FIN_WAIT → CLOSED. This enables accurate connection counting and detection of incomplete handshakes (potential port scans).

📁 File I/O

📖 Streaming PCAP Reader NOT IN WEB VERSION

The PcapReader reads packets one at a time from disk — never loading the entire file into memory. This means you can process multi-gigabyte capture files with constant memory usage.

✏️ Output PCAP Writer NOT IN WEB VERSION

Forwarded packets are written to a new PCAP file in real time. The output file contains only the "clean" traffic — blocked packets are physically absent. You can then open this file in Wireshark or feed it into another tool.

📟 CLI Reference

Flag Description Example
--block-ip <ip> Block source IP (can be repeated) --block-ip 192.168.1.50
--block-app <app> Block application by name --block-app YouTube
--block-domain <dom> Block domain (substring match) --block-domain tiktok.com
--lbs <n> Number of Load Balancer threads (default: 2) --lbs 4
--fps <n> Fast Path threads per LB (default: 2) --fps 4
